# Reality Genie Engine Integration Plan

This document outlines the architectural strategy for integrating the local-first Reality Genie AI models directly into the Reality Engine. The goal is to allow players to use real-time prompt generation tied to the engine's Visual Lambda Calculus, dynamically spawning models as they type (e.g., typing "water" creates a water voxel chunk model).

## 1. Local-First Wasm Inference

The Reality Engine runs entirely in the browser via WebAssembly (`wasm32-unknown-unknown`). To maintain our "sovereign compute" philosophy, we cannot rely on external REST APIs for the AI.

- **Weight Loading**: The `.safetensors` model weights generated by the `reality-genie` training loop (e.g., the `DiscreteDiffusion` model) will be fetched by the frontend via standard HTTP `fetch()` requests and passed into the Engine's `WorldState`.
- **In-Engine Inference**: We will compile the `candle-core` inference code to Wasm. When a player requests a new object, the Wasm module will execute the forward pass of the `DiscreteDiffusion` model locally on the user's device (falling back to WebGPU acceleration via `candle-wgpu` or running synchronously on the CPU).
- **Voxel Chunk Instantiation**: The output of the diffusion model is a sequence of discrete tokens representing Voxel IDs. The engine will deserialize these tokens into a temporary 3D Grid (`ChunkData`) and insert it into the `WorldState`.

## 2. Visual Lambda Calculus Integration

The game's spellcasting and interaction system uses a Visual Lambda Calculus. We will expose the AI generation capabilities as new primitives within this language.

### The `GENERATE` Primitive
We will add a new primitive: `GENERATE`.
- **Syntax**: `(GENERATE "text prompt")`
- **Evaluation**: When reduced, the Engine hooks into the `execute_side_effect` fallback mechanism. It sends the string literal "text prompt" to the embedded Genie inference queue.
- **Return Value**: The primitive will resolve to a unique Memory Pointer or a special `VoxelId` reference that points to the newly instantiated chunk in `world_state.dynamic_assets`.

### The `SPAWN` Primitive
To actually place the generated item into the world, players will compose `GENERATE` with `SPAWN`.
- **Syntax**: `(SPAWN (GENERATE "water") player_pos)`
- **Evaluation**:
  1. The AI generates the "water" model and returns its pointer.
  2. `SPAWN` takes the pointer and a 3D coordinate.
  3. The Engine physically places the generated voxel chunk into the world at `player_pos`.

### Visual Look-Up Tables
The Visual Lambda graph currently visualizes terms as floating bubbles. When a player casts `(GENERATE "water")`, the variable bound to that result acts as a Visual Lookup Table. Hovering over the wire or bubble containing that pointer could render a miniature real-time preview (a rotating mini-canvas) of the AI-generated voxel model before the player commits to `SPAWN`ing it into the physical world.

## 3. Real-Time Resource Management

As benchmarked in `train.rs`, running diffusion models is resource-intensive.
- **Asynchronous Execution**: In the web browser, running a 10-second diffusion pass on the main thread will cause the UI to freeze.
- **Web Workers**: The `GenieBridge` must spawn a background Web Worker (similar to the signaling server worker) to handle the `candle` inference. The Lambda primitive `GENERATE` will temporarily yield or return a "Future/Promise" token in the calculus until the Worker returns the voxel array.
- **Caching**: The Engine will maintain a `HashMap<String, ChunkData>` cache. If a player types "water" twice, the engine immediately returns the cached pointer rather than re-running the diffusion model.
